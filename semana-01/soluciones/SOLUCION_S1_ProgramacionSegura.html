Ej. 1: Validación de longitud mínima
if (password.length() < 8) {
    System.out.println("Error: contraseña demasiado corta");
}
Justificación: NIST SP 800-63B recomienda mínimo 8 caracteres. Se acepta cualquier valor ≥ 8.
Ej. 2: Verificar presencia de dígito
if (Character.isDigit(c)) {
    tieneNumero = true;
    break;
}
Método: Character.isDigit(char) — retorna true si el carácter es un dígito del 0 al 9.
Ej. 3: Verificar presencia de mayúscula
if (Character.isUpperCase(c)) {
    tieneMayuscula = true;
    break;
}
Método: Character.isUpperCase(char) — retorna true si el carácter es una letra mayúscula A-Z.
Ej. 4: Validación completa con regex
if (pass == null || pass.length() < 8)  return false;
boolean num = pass.matches("[^0-9]*[0-9].*");
boolean may = pass.matches("[^A-Z]*[A-Z].*");
boolean sim = pass.matches(".*[!@#$%^&*].*");
Propiedad: length() sin paréntesis en arrays, CON paréntesis en String. Método: matches(String regex) — evalúa la cadena completa contra la expresión regular.
Ej. 5: Validación de correo con Pattern/Matcher
Pattern patron = Pattern.compile(
    "^[\\w._%+-]+@[\\w.-]+\\.[a-zA-Z]{2,}$");
Matcher m = patron.matcher(correo);
if (m.matches()) { ... }
Método de Matcher: matches() verifica que TODA la cadena coincida con el patrón. Acepte también find() si el patrón está correctamente anclado con ^ y $.

Ej. 6: Validación de usuario alfanumérico
String usuario = sc.nextLine().trim()();
if (!usuario.matches("^[a-zA-Z0-9_]{4,20}$")) { ... }
Métodos: trim() elimina espacios en blanco al inicio y al fin. matches(regex) evalúa la cadena completa. El patrón restringe a caracteres alfanuméricos y guion bajo entre 4 y 20 caracteres.
Ej. 7: Sanitización — eliminar caracteres peligrosos
String limpia = entrada
    .replace("<","").replace(">","")
    .replace("'","").replace("\"","")
    .replace(";","").replace("--","");
Método: replace(CharSequence target, CharSequence replacement) — reemplaza todas las ocurrencias de una cadena. Nota: en contextos reales usar bibliotecas como Apache Commons Text o OWASP ESAPI.
Ej. 8: Control de acceso con switch
switch (rol) {
    case "ADMIN":
        System.out.println("Acceso total");
        break;
    case "VENDEDOR":
        System.out.println("Acceso a ventas");
        break;
    default:
        System.out.println("Rol no reconocido: acceso denegado");
}
Estructura: switch — break para salir de cada caso — default para el caso por defecto. Sin break ocurre fall-through (error de seguridad).
Ej. 9: Bloqueo por intentos fallidos
final int LIMITE = 3;
if (pass.equals("ClaveCorrecta")) { ... break; }
bloqueado = true;
Valor: OWASP recomienda 3-5 intentos. equals() compara contenido (no referencia). Acepte valores entre 3 y 5. En producción el bloqueo debe ser persistente (base de datos), no solo en memoria.
Ej. 10: Contraseña no igual ni contenga al usuario
if (password.equals(usuario)) {
    System.out.println("La contraseña no puede ser igual al usuario");

SOLUCIONES — ANÁLISIS DE CÓDIGO
Cada fragmento completo bien analizado vale 0.4 pts → 10 × 0.4 = 4.0 pts
✓ Análisis
Fragmento 1[Contraseña en texto plano]⚠ Inseguro
1. Problema: Contraseña hardcodeada (hardcoded password) en texto plano directamente en el código fuente.
2. Riesgo: Cualquier persona que acceda al código fuente, el repositorio Git o el binario compilado puede obtener la contraseña real. Viola el principio de separación de credenciales y secretos. Categoría OWASP: A07 – Fallas de identificación y autenticación.
3. Corrección: Las credenciales deben almacenarse como hash BCrypt en base de datos. En tiempo de ejecución: obtener la contraseña del usuario, aplicarle hash y comparar con el hash almacenado usando BCrypt.checkpw(inputPass, storedHash). Nunca almacenar ni comparar contraseñas en texto plano.
Fragmento 2[Validación de entrada]✓ Seguro
1. Confirmación: El código ES seguro. Valida: (a) que el parámetro no sea null, (b) que no esté vacío tras aplicar trim(), y (c) que cumpla la expresión regular que restringe a caracteres alfanuméricos y guion bajo con longitud entre 4 y 20 caracteres.
2. Principio: Validación de entrada (Input Validation) — rechazar todo lo que no cumple el formato esperado. Defensa en profundidad y principio de lista blanca (allowlist) sobre lista negra.
3. Mejoras posibles: Agregar límite de longitud máxima explícito en el método (ya lo hace el regex). Considerar también verificar que no contenga palabras reservadas del sistema. En general el código es correcto para la validación de nombre de usuario.
Fragmento 3[Ausencia total de validación]⚠ Inseguro
1. Problema: Los datos ingresados por el usuario (user y email) se pasan directamente a registrarUsuario() sin ningún tipo de validación, sanitización ni verificación de formato.
2. Riesgo: Permite inyección de caracteres maliciosos, registro con datos inválidos (emails falsos, usuarios vacíos), y potencial SQL Injection o XSS si los datos llegan a base de datos o interfaz web sin tratamiento. OWASP A03 – Injection.
3. Corrección: Antes de llamar a registrarUsuario() validar: (a) usuario no vacío y alfanumérico, (b) email con regex de formato válido, (c) longitud máxima para ambos campos. Aplicar sanitización básica eliminando caracteres especiales peligrosos.
Fragmento 4[Control de acceso por rol]✓ Seguro
1. Confirmación: El código ES seguro. El rol se obtiene de un objeto de sesión del sistema (sesion.obtenerRol()), NO del input del usuario, lo que evita que el atacante elija su propio rol. El default deniega acceso a roles desconocidos.
2. Principio: Control de acceso basado en roles (RBAC) con mínimo privilegio. El rol proviene de una fuente confiable (sesión del servidor). OWASP A01 – Broken Access Control (aquí correctamente implementado).
3. Mejoras: Usar un enum en lugar de String para los roles (evita errores tipográficos). Agregar auditoría/logging del acceso. Verificar también que la sesión sea válida y no haya expirado antes de leer el rol.

Fragmento 5[Construcción SQL con concatenación]⚠ Inseguro
1. Problema: La variable usuario ingresada por el usuario se concatena directamente en la cadena SQL. Esto es SQL Injection clásico. Un atacante puede ingresar: ' OR '1'='1 para acceder sin credenciales, o '; DROP TABLE usuarios; -- para destruir datos.
2. Riesgo: Acceso no autorizado a todos los registros, exfiltración de datos sensibles, modificación o eliminación de la base de datos. Clasificado como OWASP A03:2021 – Injection. Es la vulnerabilidad más crítica en aplicaciones con base de datos.
3. Corrección con PreparedStatement:
PreparedStatement ps = conn.prepareStatement("SELECT * FROM usuarios WHERE nombre = ?");
ps.setString(1, usuario);
ResultSet rs = ps.executeQuery();
Los parámetros se tratan como datos, nunca como código SQL ejecutable.
Fragmento 6[Mensaje de error revelador]⚠ Inseguro
1. Problema: Los mensajes de error revelan explícitamente si el usuario existe o no, y si la contraseña es incorrecta. Esto permite a un atacante realizar enumeración de usuarios: puede probar nombres de usuario sistemáticamente hasta encontrar uno válido.
2. Riesgo: Facilita ataques de fuerza bruta dirigidos y ataques de credential stuffing, ya que el atacante sabe exactamente qué campo es correcto. OWASP A07 – Fallas de identificación y autenticación.
3. Corrección: Usar siempre un mensaje genérico independientemente de cuál campo falló:
if (!usuarioExiste || !contrasenaCorrecta) {
System.out.println("Credenciales inválidas. Intente nuevamente.");
}
El tiempo de respuesta también debe ser similar en ambos casos (mismo tiempo de procesamiento).
Fragmento 7[Bloqueo por intentos fallidos]✓ Seguro
1. Confirmación: El código ES seguro. Implementa un mecanismo de bloqueo tras 3 intentos fallidos con la variable bloqueado, el ciclo while se detiene cuando se supera el límite o cuando las credenciales son correctas.
2. Principio: Rate limiting / Account lockout — protege contra ataques de fuerza bruta y credential stuffing. OWASP recomienda bloqueo temporal tras 3-5 intentos fallidos. Principio de defensa en profundidad.
3. Mejoras: El bloqueo debería ser persistente (guardarlo en base de datos), no solo en memoria (al reiniciar el programa el contador se pierde). Agregar desbloqueo temporal con tiempo de espera (ej: 15 minutos) en lugar de bloqueo permanente. Registrar en log de auditoría los intentos fallidos con fecha/hora e IP.

Fragmento 8[SQL Injection en consulta de productos]⚠ Inseguro
1. Problema: Igual que el fragmento 5: la variable cat se concatena directamente en la cadena SQL. Es SQL Injection. Un atacante puede ingresar ' UNION SELECT usuario, contrasena FROM usuarios -- para extraer datos de otra tabla.
2. Riesgo: Extracción de datos de cualquier tabla de la base de datos, incluyendo tablas de usuarios y contraseñas. OWASP A03 – Injection. Este tipo de ataque es automatizable con herramientas como sqlmap.
3. Corrección: Usar PreparedStatement con parámetro ?:
PreparedStatement ps = conn.prepareStatement(
"SELECT nombre, precio FROM productos WHERE categoria = ? ORDER BY precio ASC");
ps.setString(1, cat); ResultSet rs = ps.executeQuery();
Fragmento 9[Validación robusta de contraseña]✓ Seguro
1. Confirmación: El código ES seguro. Verifica null, longitud mínima de 8, presencia de mayúscula, dígito y símbolo especial mediante expresiones regulares. La verificación de null al inicio previene NullPointerException.
2. Principio: Validación de complejidad de contraseña. Sigue las recomendaciones de NIST SP 800-63B para contraseñas: mínimo de caracteres, diversidad de tipos. El operador && en cadena es eficiente (short-circuit evaluation).
3. Mejoras: Agregar verificación de longitud máxima (ej: máximo 64 caracteres para prevenir ataques DoS con contraseñas enormes). Verificar que la contraseña no esté en una lista de contraseñas comunes (diccionario de contraseñas débiles conocidas). Considerar usar una biblioteca especializada como ZXCVBN para evaluar fortaleza real.
Fragmento 10[Rol definido por el propio usuario]⚠ Inseguro
1. Problema: El rol es leído directamente del input del usuario con sc.nextLine(). Cualquier usuario puede escribir "ADMIN" y obtener acceso administrativo total. No hay verificación de identidad previa ni autenticación.
2. Riesgo: Escalada de privilegios sin autenticación. Un usuario cualquiera puede acceder al panel administrativo simplemente escribiendo "ADMIN". OWASP A01:2021 – Broken Access Control. Es uno de los errores más graves en control de acceso.
3. Corrección: El rol debe obtenerse de la sesión autenticada del servidor, nunca del input del usuario. Ejemplo correcto:
String rol = sesionAutenticada.getRolDelUsuario(); // del servidor
if ("ADMIN".equals(rol) && sesionAutenticada.estaActiva()) {
Adicionalmente: verificar siempre que la sesión sea válida y no haya expirado.

import java.util.Scanner;

public class LoginSeguro {

    // ── Constantes de seguridad ──────────────────────────────────
    private static final int    MIN_LENGTH  = 8;
    private static final int    MAX_INTENTOS = 3;

    // ── Punto de entrada ─────────────────────────────────────────
    public static void main(String[] args) {
        Scanner sc = new Scanner(System.in);

        System.out.print("Nombre de usuario: ");
        String usuario = sc.nextLine().trim();

        int intentosFallidos = 0;
        boolean accesoConcedido = false;

        while (intentosFallidos < MAX_INTENTOS) {
            System.out.print("Contraseña: ");
            String contrasena = sc.nextLine();

            if (validarContrasena(contrasena)) {
                accesoConcedido = true;
                System.out.println("✓ Acceso concedido. Bienvenido, " + usuario);
                break;
            } else {
                intentosFallidos++;
                int restantes = MAX_INTENTOS - intentosFallidos;
                if (restantes > 0) {
                    System.out.println("✗ Contraseña no válida. Intentos restantes: " + restantes);
                } else {
                    System.out.println("⛔ Cuenta bloqueada. Demasiados intentos fallidos.");
                }
            }
        }
        sc.close();
    }

    // ── Validación completa de contraseña ───────────────────────
    public static boolean validarContrasena(String pass) {
        if (pass == null || pass.length() < MIN_LENGTH) {
            System.out.println("  → Mínimo " + MIN_LENGTH + " caracteres requeridos.");
            return false;
        }

        boolean tieneNumero    = false;
        boolean tieneMayuscula = false;

        for (char c : pass.toCharArray()) {
            if (Character.isDigit(c))     tieneNumero    = true;
            if (Character.isUpperCase(c)) tieneMayuscula = true;
        }

        if (!tieneNumero) {
            System.out.println("  → La contraseña debe contener al menos un número.");
            return false;
        }
        if (!tieneMayuscula) {
            System.out.println("  → La contraseña debe contener al menos una letra mayúscula.");
            return false;
        }
        return true;
    }
}

H.3 — Análisis de 3 líneas clave (solución modelo)

N° Línea de código Explicación de seguridad
1 pass.length() < MIN_LENGTH Valida la longitud mínima de la contraseña (8 caracteres). Las contraseñas cortas son vulnerables a ataques de fuerza bruta. NIST SP 800-63B recomienda mínimo 8 caracteres. El uso de una constante (MIN_LENGTH) facilita el mantenimiento.
2 Character.isDigit(c) / isUpperCase(c) Verifica la complejidad de la contraseña: presencia de dígitos y mayúsculas. Las contraseñas complejas resisten mejor los ataques de diccionario y fuerza bruta. Usar métodos de Character garantiza compatibilidad con caracteres Unicode.
3 while (intentosFallidos < MAX_INTENTOS) Implementa el mecanismo de bloqueo por intentos fallidos. Después de MAX_INTENTOS (3) intentos incorrectos, el acceso queda bloqueado. Protege contra ataques de fuerza bruta y credential stuffing automatizados.
